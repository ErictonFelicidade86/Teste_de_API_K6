# K6 — Testes de Performance e Carga

## Visão Geral

O K6 é responsável pelos testes de performance e carga da FakeStore API. São 4 tipos de teste, cada um com um objetivo específico, cobrindo desde a validação básica até simulação de picos extremos de tráfego.

**API alvo:** `https://fakestoreapi.com`

---

## Estrutura

```
k6/
├── tests/
│   ├── smoke-test.js   # Validação básica — 1 VU / 30s
│   ├── load-test.js    # Carga normal — 50 VUs / 30s
│   ├── stress-test.js  # Stress progressivo — até 200 VUs em rampa
│   └── spike-test.js   # Pico repentino — até 500 VUs em 10s
└── Dockerfile
```

---

## Tipos de Teste

### Smoke Test (`smoke-test.js`)
Validação mínima para garantir que a API está respondendo corretamente. Roda com apenas 1 usuário virtual durante 30 segundos.

**Quando usar:** antes de qualquer outra execução, para confirmar que a API está operacional.

**Configuração:**
- VUs: 1
- Duração: 30s
- p(95) < 1000ms | erros < 1%

---

### Load Test (`load-test.js`)
Simula a carga normal esperada da API com múltiplos usuários simultâneos. Inclui métricas customizadas por grupo de endpoints.

**Quando usar:** validação de comportamento sob uso típico.

**Configuração:**
- VUs: 50
- Duração: 30s
- p(95) < 500ms | erros < 5%
- Métricas customizadas: `product_duration`, `login_duration`, `cart_duration`, `requests_total`

---

### Stress Test (`stress-test.js`)
Aumenta a carga gradualmente em etapas para identificar o ponto de saturação da API.

**Quando usar:** identificar o limite de capacidade da API.

**Configuração (stages):**

| Etapa | Duração | VUs |
|-------|---------|-----|
| Aquecimento | 1 min | 0 → 50 |
| Estável | 2 min | 50 |
| Subida | 1 min | 50 → 100 |
| Estável | 2 min | 100 |
| Pico | 1 min | 100 → 200 |
| Estável | 2 min | 200 |
| Resfriamento | 1 min | 200 → 0 |

- p(95) < 1500ms | erros < 5%

---

### Spike Test (`spike-test.js`)
Simula um pico repentino de tráfego, como uma promoção ou evento viral.

**Quando usar:** validar resiliência da API sob carga inesperada e abrupta.

**Configuração (stages):**

| Etapa | Duração | VUs |
|-------|---------|-----|
| Normal | 10s | 5 |
| Spike | 10s | 5 → 500 |
| Pico mantido | 1 min | 500 |
| Retorno | 10s | 500 → 5 |
| Recuperação | 30s | 5 |
| Finalização | 10s | 0 |

- p(95) < 2000ms | erros < 10%

## Como Executar

### Localmente (k6 instalado)

```bash
# Smoke test
k6 run k6/tests/smoke-test.js

# Load test
k6 run k6/tests/load-test.js

# Stress test
k6 run k6/tests/stress-test.js

# Spike test
k6 run k6/tests/spike-test.js
```

### Via Docker

```bash
# Build da imagem
docker build -t k6-fakestore ./k6

# Rodar teste específico
docker run --rm k6-fakestore run smoke-test.js
docker run --rm k6-fakestore run load-test.js
docker run --rm k6-fakestore run stress-test.js
docker run --rm k6-fakestore run spike-test.js
```

## Pipeline de Execução

### GitHub Actions
Executa apenas o **smoke test** a cada push na branch `main`. Se passar, faz o build e push da imagem Docker para o `ghcr.io`.

```
push na main → smoke test → build + push ghcr.io
```

### Jenkins
Executa **todos os testes** em sequência, agendado via cron (08h e 20h). Cada stage roda de forma independente — uma falha não interrompe os demais.

```
Smoke → Load → Stress → Spike
```
